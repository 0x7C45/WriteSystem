# AI 绘制解释型图表 MVP 评价体系

> 本文档是一套最小可用的评价标准，用于判断 AI 生成的流程图、框架图、概念图、科研示意图等解释型图表的质量。
> 核心对象是**解释型图表**——主要用来说明概念、系统组成、流程运行、框架分层、机制发生、研究方案推进、复杂问题拆解的图表。
> 第一目标不是艺术表现，而是：**让读者更快、更准确地理解一个复杂对象。**

---

## 适用图表类型

| 类型 | 用途 | 评价重点 |
|------|------|---------|
| 流程图 | 解释步骤、顺序、执行路径 | 起点终点、步骤顺序、分支条件、反馈回路 |
| 框架图 | 解释系统/理论/方案的组成 | 层级合理性、模块完整性、主次结构 |
| 架构图 | 解释系统或技术方案的组成和关系 | 输入处理输出完整、模块职责、数据流向、系统边界 |
| 概念图 | 解释概念的定义、组成、边界和关系 | 概念拆解准确、避免概念偷换、区分核心与相关 |
| 关系图 | 解释多对象间的连接、影响、协作或依赖 | 关系类型清楚、箭头含义明确、区分因果/依赖/包含 |
| 机制图 | 解释某个过程为什么发生、如何发生 | 触发因素、过程链条、反馈机制、区分原因/过程/结果 |
| 科研示意图 | 解释研究对象、实验流程、理论机制 | 科研事实准确、方法链条完整、符合学术表达 |

**不适用**：科研数据图表（折线图、柱状图、散点图等）需要额外补充数据真实性、统计规范性、坐标单位、学科表达规范等维度。

## 5 个核心维度

### 维度一：语义准确性（Semantic Accuracy）

**核心问题**：这张图有没有画错？

检查内容：
- 概念名称是否正确
- 节点之间的关系是否正确
- 箭头方向是否正确
- 因果关系有没有被误画
- 并列关系有没有被误画成上下级
- 层级关系有没有被误画成流程
- 输入、输出、反馈有没有被误解
- 不确定内容有没有被画成确定结论
- 图号（图X-X）和图名是否错误嵌入图表（应属论文排版，不得出现在图内）

**如果这一项不合格，后面的优化没有意义。**

### 维度二：结构逻辑（Structural Logic）

**核心问题**：这张图的关系组织是否合理？

检查内容：
- 图表类型选择是否正确（流程图 vs 框架图 vs 架构图 vs 关系图）
- 主线是否清楚
- 分支是否合理
- 层级是否明确
- 并列项是否真的并列
- 上下游关系是否正确
- 是否存在反馈回路
- 是否有清晰的起点和终点
- 是否把不同类型的信息混在同一层级

### 维度三：抽象粒度（Abstraction Granularity）

**核心问题**：这张图画得太粗、太细，还是刚好？

检查内容：
- 图表是否只剩空泛大词
- 图表是否塞入过多细节
- 同一层级的节点是否粒度一致
- 是否把战略层、业务层、技术层混在一起
- 是否把代码细节放进概念图
- 是否把结论性概念放进执行流程
- 是否适合当前读者和当前使用场景

### 维度四：视觉清晰度（Visual Clarity）

**核心问题**：这张图能不能被轻松看懂？

检查内容：
- 字体是否足够大
- 节点是否重叠
- 箭头是否混乱
- 连线是否交叉过多
- 模块间距是否合理
- 分区是否清楚
- 阅读方向是否明确
- 颜色是否辅助理解而不是制造干扰
- 缩小后是否仍能看清

### 维度五：可修改性（Editability）

**核心问题**：这张图后续能不能快速迭代？

检查内容：
- 是否有可编辑源文件
- 是否能修改单个节点
- 是否能修改箭头关系
- 是否能调整层级
- 是否能复用到其他图表
- 是否能版本管理
- 是否避免只能生成一张不可编辑图片

## 评分标准

### 单维度评分

| 维度 | 0 分 | 3 分 | 5 分 |
|------|------|------|------|
| 语义准确性 | 概念和关系明显错误 | 基本正确，有少量偏差 | 完全忠实表达原意 |
| 结构逻辑 | 结构混乱，类型错误 | 主线基本清楚 | 结构类型准确，层级清晰 |
| 抽象粒度 | 太粗、太细或混乱 | 基本适合当前场景 | 粒度稳定，层级一致 |
| 视觉清晰度 | 难以阅读 | 基本能看懂 | 清楚、顺畅、重点突出 |
| 可修改性 | 只能重画 | 可局部修改 | 可编辑、可复用、可版本管理 |

### 总分判断

| 总分 | 判断 |
|-----:|------|
| 0–10 | 不可用，建议重画 |
| 11–16 | 方向可保留，但需要大改 |
| 17–21 | 基本可用，需要优化 |
| 22–25 | 可交付，可作为模板沉淀 |

## 常见结构反模式清单

> 维度二（结构逻辑）和维度三（抽象粒度）的**反面案例**。生成图表后对照检查，出现以下任一反模式时，对应维度通常 ≤3 分。

### R1. 扁平辐射反模式

**识别特征**：1 个中心节点 + N 个并列叶子节点（通常 ≤7 节点），无二级展开，无 subgraph 分组。

**为什么扣分**：结构扁平、粒度粗糙，信息密度不足。

**修复建议**：
- 若内容是分类体系 → 使用分类图模板（flowchart + 嵌套 subgraph），增加"分类标准"中间层
- 若内容是概念拆解 → 使用 mindmap 语法，自动渲染层级递进
- 为每个叶子节点补充 2-3 个子项，提升粒度到 ≥3 级

**对应维度**：结构逻辑（层级不明确）+ 抽象粒度（信息稀疏）

### R2. 单线链反模式

**识别特征**：≤5 个节点线性排列（A→B→C→D→E），无并行分支，无阶段分组，无反馈环。

**为什么扣分**：线性链丢失了并行关系、依赖关系和阶段差异，语义不完整。

**修复建议**：
- 若内容是技术路线/阶段规划 → 使用 timeline 语法（section 阶段分组）
- 若存在并行任务 → 增加分叉/汇合路径
- 用 subgraph 或 cluster 将节点按阶段分组

**对应维度**：结构逻辑（主线单一）+ 语义准确性（关系遗漏）

### R3. 信息稀疏反模式

**识别特征**：总节点数 < 8，且节点间无方向性依赖关系（只有并列或无向连接）。

**为什么扣分**：信息密度不足以支撑独立图表的存在价值。

**修复建议**：
- 为关键节点增加量化指标（数量、比例、周期等）
- 引入节点间的前后依赖箭头（→），替代并列连接（---）
- 若素材天然简单（如仅 5 个并列项），使用 mindmap 而非 flowchart

**对应维度**：抽象粒度（太粗）

### R4. 装饰节点反模式

**识别特征**：图中存在"设计理念""核心思想""模块化设计"等非信息节点，不传递具体技术内容，仅充当视觉填充。

**为什么扣分**：占用图面空间但不增加信息量，降低信息密度。

**修复建议**：
- 删除装饰节点，将空间让给有实际内容的节点
- 或将装饰性文字改为图标题/图注，而非图中节点

**对应维度**：语义准确性（无实际语义）+ 抽象粒度（粒度浪费）

### 最低结构标准

生成图表时应达到以下最低标准：

| 工具 | 最低节点数 | 最低结构深度 |
|------|-----------|-------------|
| Mermaid flowchart | ≥10 节点 | ≥2 层 subgraph |
| Mermaid mindmap | ≥10 节点 | ≥3 层层级 |
| Mermaid timeline | ≥3 个 section | 每个 section ≥1 个 task |
| Mermaid sequenceDiagram | ≥3 个 participant | ≥3 个交互阶段 |
| Graphviz | ≥8 节点 | ≥1 个 cluster 分组 |

若素材内容天然简单，优先使用 mindmap 或 timeline，而非强行用 flowchart 扩充。

## 适用生产阶段

| 阶段 | 用途 |
|------|------|
| 构思阶段 | 判断内容适合画成什么图、目标读者是谁、应保留哪些信息 |
| AI 生成阶段 | 用 5 个维度约束提示词，防止 AI 乱改原意或选错图表类型 |
| 初稿审查阶段 | 判断图表能否继续改、哪个维度问题最大、下一轮修改重点 |
| 迭代修改阶段 | 每轮修改后检查语义是否漂移、结构是否被破坏、粒度是否失控 |
| 交付阶段 | 判断图表是否能进入 PPT/论文/网页/答辩等实际材料 |

---

## 修改优先级

| 优先级 | 内容 |
|--------|------|
| 必须改 | 先明确图表类型 |
| 必须改 | 保证语义准确 |
| 必须改 | 控制抽象粒度 |
| 应该改 | 输出可编辑格式 |
| 可后置 | 美观风格、交互性、品牌化 |
| 不值得做 | 一开始建立 15 个以上复杂维度 |

---

## 使用流程

```
内容 → 判断图表类型 → AI 生成图表 → 用 5 个维度评估 → 找出问题 → 重新提示 AI 修改 → 得到可交付图表
```

## 后续扩展方向

- 美观度（配色、排版、视觉吸引力）
- 交互性（悬停提示、缩放、可点击导航）
- 可移植性（跨平台查看、输出格式多样性）
- 可扩展性（图表复杂度增长时的表现）
- 数据规范性（仅适用于科研数据图表）
